# Agent Spec

Version: 2 - Mitigation-first design

## Overview

NoiceCance is an agent-assisted acoustic mitigation prototype for people exposed to chronic transportation noise. The first target users are residents living near two high-noise environments:

1. A two-way six-lane intersection with braking, horns, tire noise, engine rumble, and nighttime traffic peaks.
2. Homes near an airport with intermittent aircraft takeoff and landing noise, low-frequency rumble, and high peak events.

The project is not a universal active-noise-control generator. It is a noise-mitigation decision agent that determines when active noise control is appropriate, when it is unsafe or ineffective, and what alternative controls should be used.

The project is designed as an offline-first acoustic planning system with an optional connected agent layer. The real-time acoustic pipeline should not depend on an LLM or internet connection. Agents operate at the supervisory layer: they interpret the user's noise problem, analyze recordings or simulated profiles, select safe mitigation strategies, call tools exposed through MCP, and explain the resulting plan.

The capstone submission should demonstrate a practical path from user complaint to a structured mitigation plan:

- User describes a noise problem or provides a short recording.
- The system extracts privacy-preserving acoustic features.
- A multi-agent planner classifies which controls are suitable.
- A simulator visualizes noise sources, target quiet zones, microphone placement, speaker placement, and expected effects when active control is plausible.
- The system exports `mitigation_plan.json`.
- If the noise profile is suitable for local low-frequency ANC, the mitigation plan may include an `anc_policy` section for a future microphone and speaker array.
- If the noise profile is dominated by high-frequency, impulsive, or complex reverberant noise, the plan should explicitly disable ANC and recommend passive, engineering, behavioral, or hearing-protection controls instead.

## Example Use Cases

### Six-Lane Intersection Resident

Input:

> I live on the second floor facing a two-way six-lane intersection. At night I hear braking, horns, motorcycles, and a low engine rumble. I want to sleep but still hear alarms or emergency sounds.

Expected output:

- Identifies likely broadband traffic noise, impulsive horn events, and low-frequency rumble.
- Recommends a sleep-focused mitigation plan.
- Preserves alarms, sirens, smoke detectors, and urgent human speech.
- Suggests passive controls for mid/high frequencies and local quiet-zone ANC only for plausible low-frequency components.
- Suggests microphone and speaker placement near the window and the bed-side target quiet zone only when ANC is suitable.
- Generates `mitigation_plan.json`, optionally including an `anc_policy` section with conservative limits, pass-through sounds, and target attenuation bands.

### Airport-Adjacent Resident

Input:

> I live near an airport. Several times per hour, aircraft create a loud low-frequency wave that wakes my family. We need a plan that works at night.

Expected output:

- Identifies intermittent high-energy events with low-frequency components.
- Recommends event-aware mitigation, prediction windows, passive controls, and non-ANC supporting actions.
- Explains why full-room cancellation is unrealistic without calibrated hardware.
- Generates a conservative mitigation plan focused on a limited quiet zone near the bed if low-frequency ANC is plausible.
- Blocks ANC for high-frequency, impulsive, or poorly predictable aircraft noise components that cannot be reliably cancelled.

### Unsuitable ANC Scenario

Input:

> My apartment is near a workshop and most of the annoying noise is sharp, high-pitched, and unpredictable. Can the system cancel it with a speaker?

Expected output:

- Identifies high-frequency and impulsive noise as poor candidates for open-air ANC.
- Sets `anc_policy.enabled` to `false`.
- Recommends passive insulation, sealing air gaps, absorber placement, administrative controls, or certified hearing protection depending on the setting.
- Explains that attempting to cancel these sounds with an active speaker may fail or create louder hot spots.

### Recording-Based Analysis

Input:

> A short audio recording captured locally from the user's laptop or phone.

Expected output:

- Extracts feature summaries such as approximate loudness, spectrum bands, peak events, periodicity, and dominant frequency regions.
- Does not store or upload raw audio by default.
- Uses features to select or refine a mitigation strategy.
- Marks ANC suitability as `recommended`, `partial`, or `not_recommended`.

## Tools Required

### MCP Tools

The MCP server should expose deterministic tools that the agents can call:

- `analyze_noise_profile`
  - Input: scenario metadata and optional extracted audio features.
  - Output: noise category, dominant bands, peak behavior, and confidence notes.

- `assess_control_suitability`
  - Input: scenario metadata, extracted audio features, user goals, and hardware assumptions.
  - Output: suitability scores for near-field ANC, passive insulation, masking sound, hearing protection, and engineering controls.

- `simulate_sound_field`
  - Input: room layout, noise source direction, quiet zone, microphone placement, speaker placement, and candidate policy.
  - Output: estimated before/after attenuation map and caveats.

- `generate_mitigation_plan`
  - Input: scenario, user goals, safety constraints, and simulation results.
  - Output: `mitigation_plan.json` with recommended controls, blocked controls, evidence, caveats, and optional ANC settings.

- `generate_anc_policy`
  - Input: scenario, user goals, safety constraints, and simulation results.
  - Output: optional `anc_policy` object for a future deployable ANC unit.
  - Constraint: only enabled when suitability checks confirm that local low-frequency ANC is plausible.

- `check_safety_limits`
  - Input: candidate policy.
  - Output: safety decision, warnings, blocked settings, and required pass-through sounds.

- `render_mitigation_report`
  - Input: full analysis and selected policy.
  - Output: user-facing explanation suitable for the web demo and writeup.

### Local Acoustic Analyzer

The analyzer should run locally and avoid saving raw recordings by default. It should derive privacy-preserving features such as:

- Approximate loudness envelope.
- Frequency-band energy summaries.
- Peak event count and timing.
- Low-frequency dominance.
- Impulsive event indicators.

### Web Demo

The web demo should support:

- Scenario selection for six-lane intersection and airport-adjacent housing.
- Free-text noise complaint entry.
- Optional local recording upload.
- Agent-generated mitigation plan.
- Offline mode using built-in scenario templates.
- Visualization of noise source, quiet zone, microphones, speakers, and before/after estimates.
- Export of `mitigation_plan.json`.
- Optional export of an `anc_policy` subsection only when active control is suitable.

## Constraints & Safety Rules

- The system must not claim to provide medical diagnosis.
- The system must not replace professional occupational safety, acoustic engineering, or hearing-protection advice.
- The system must not claim that a laptop speaker can provide real room-scale cancellation for traffic or aircraft noise.
- The system must not claim that ultrasonic transducers can directly cancel audible traffic or aircraft noise.
- The system must not present ANC as a universal solution for high-frequency, impulsive, unpredictable, or strongly reverberant noise.
- The real-time audio path should be local-first and must not require LLM inference.
- Raw audio should not be uploaded or stored by default.
- The system should preserve alarms, sirens, smoke detectors, urgent human speech, and other safety-critical sounds.
- ANC amplitude, output volume, and cancellation profiles must have conservative safety limits.
- The agent must explain when a scenario is outside the capability of simulated or consumer-grade equipment.
- The agent must be able to refuse ANC and recommend non-ANC controls when active cancellation is physically unsuitable.
- The project must not include API keys, passwords, tokens, or private credentials in code or documentation.

## Architecture & Sub-Agents

### User Intent Agent

Interprets the user's noise complaint and extracts structured goals:

- Sleep protection.
- Work or study focus.
- Family or child comfort.
- Safety-critical sound preservation.
- Privacy preference.
- Hardware availability.

### Acoustic Scene Agent

Maps the complaint and extracted audio features to an acoustic scene:

- Intersection traffic.
- Airport takeoff or landing events.
- Low-frequency rumble.
- Impulsive peaks.
- Intermittent versus continuous noise.

### Policy Planning Agent

Chooses a mitigation strategy and calls MCP tools:

- Selects target quiet zone.
- Chooses microphone and speaker placement assumptions.
- Decides whether ANC should be enabled, partially enabled, or blocked.
- Generates a complete mitigation plan.
- Generates a candidate ANC policy only when suitable.
- Requests simulation when a candidate active strategy exists.
- Iterates if safety checks reject the policy.

### Safety & Privacy Agent

Audits the plan:

- Blocks unsafe output levels.
- Requires pass-through for alarms and emergency cues.
- Confirms raw audio is not retained by default.
- Rejects ultrasonic or active-control claims that are not supported by the noise profile.
- Adds caveats for unrealistic or unsupported claims.

### Report Agent

Produces user-facing and judge-facing explanations:

- Plain-language mitigation summary.
- Technical architecture explanation.
- `mitigation_plan.json` explanation.
- Optional `anc_policy` explanation when included.
- Limitations and next steps.

## Deployment Preference

### Phase 1: Local Web Demo

The first deliverable should be a local web demo with clear README instructions. It should run on a laptop and use built-in scenarios even when no network or LLM access is available.

### Phase 2: Containerized Demo

After the local demo works, add a containerized deployment path if approved. Docker or compose files are deployment-related and require explicit approval before editing under this workspace's rules.

### Phase 3: Deployable Mitigation Plan

The project should export `mitigation_plan.json` as the stable interface. This plan may include an `anc_policy` section when near-field low-frequency active control is plausible. The same plan can also recommend passive insulation, room layout changes, masking sounds, or hearing protection when ANC is not appropriate.

The capstone prototype should not pretend to be a calibrated hardware unit. It should simulate the target and clearly distinguish plausible local quiet-zone ANC from unsupported whole-room or high-frequency cancellation.

## Success Criteria

- Demonstrates at least two transportation-noise scenarios: six-lane intersection and airport-adjacent home.
- Accepts a user complaint and returns a structured mitigation plan.
- Supports optional recording-based feature analysis without retaining raw audio by default.
- Shows a multi-agent ADK design in code or clearly documented architecture.
- Exposes acoustic analysis and policy operations through MCP-style tools.
- Generates `mitigation_plan.json`.
- Correctly disables ANC for high-frequency, impulsive, or unsuitable acoustic profiles.
- Generates a conservative `anc_policy` only for plausible low-frequency local quiet-zone scenarios.
- Includes clear safety and privacy constraints.
- Provides a local demo and README that judges can run or inspect.
- Includes diagrams or visualizations suitable for the Kaggle Writeup and video.
